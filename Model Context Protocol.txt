Tripartite Architecture

MCP host (Coordinator)
AI application (IDE,Agent,Claude desktop)

MCP client (Connector)
connected to server via JSON_RPC 2.0 protocol (Data layer)

MCP Server (Context Provider)

In our example:-
MCP server is Paypal
MCP client is Google gemini
MCP Host is AI Agent

MCP client to server connected via 2 ways (Transport layers):-
1. Local (stdio transport) - best local security and high performance
   Lifespan: persistent till host app is open
   
2. Remote (SSE/HTTP transport) SSE deprecated - distributed systems - HTTP POST for client to server messages
   Lifespan: transient/session based (full handshake and re-initialization required)


Data layer JSON_RPC primitives:-
Resources - read only data (GET endpoints) "method": "resources/listcity"
Tools     - executable functions (POST endpoints) "method": "tools/getWeather"
Prompts   - reusable data templates for LLM interactions "method": "prompts/getcityweather"

doc strings contain information about the tool in MCP.

In case a dependency in required in implementation, an additional information needs to be given, 
then it cannot go in doc string, rather it should be given in prompt itself.

JSON RPC request payload example:

{
  "jsonrpc": "2.0",
  "method": "tools/getWeather",
  "params": {
  "name": "weather_current",
  "arguments":{
    "city": "London"
	}
  },
  "id": 1
}


Server implementation strategy:-

1. Low level SDK (high control less velocity)
2. FASTMCP       (most pythonic way using decorators)        
3. MCP Toolboxes (less control high velocity)

Security and Authentication using:-
OAuth 2.1
Input validation
Least priviledge policy

Authorization:-
Human in loop
Data privacy to restrict sensitive data

local : mcp docker with 8000 port
Orchestration : using Docker image on cloud run
create agent registry using cloud build and then expose it on endpoint with endpoint/mcp
